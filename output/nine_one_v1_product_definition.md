# Nine One V1 Product Definition

## What the product actually is

Nine One V1 is **a trusted padel match and improvement system for local venue communities**.

It is not a generic sports app. It is not a broad booking marketplace. It is not a fitness feed.

The product exists to make one behavior easier and richer:

**A player finds the right match, shows up, plays with people they can trust, learns something from it, and returns.**

V1 should treat every match as a living event with before, during, and after states:

| Stage | Product role |
| --- | --- |
| Before | Help the player find a match that fits their level, time, venue, and trust needs. |
| Commitment | Make payment, cancellation, attendance, and expectations clear. |
| Play | Stay mostly quiet while the real experience happens. |
| After | Turn the match into memory: result, balance, reliability, level confidence, next step. |
| Return | Recommend the next match, squad, clinic, or coach action. |

The strategic object is not a booking.

The strategic object is a **quality completed match**.

## What the user opens the app for

The player should open Nine One for one of five reasons:

| Reason | Underlying need |
| --- | --- |
| Find a match I can trust | Belonging and confidence |
| See who I am playing with | Safety and fit |
| Confirm I am expected | Commitment |
| Understand my last match | Progress |
| Know what to do next | Guidance |

The venue should open Nine One to know:

| Reason | Underlying need |
| --- | --- |
| Which sessions are filling | Court utilization |
| Where demand is building | Programming intelligence |
| Who is reliable | Community quality |
| Which players are returning | Retention |
| Which clinics or open-play sessions to run | Growth |

The coach should open Nine One to know:

| Reason | Underlying need |
| --- | --- |
| Which players need validation | Trust building |
| Which players are ready for coaching | Qualified demand |
| What patterns are emerging | Better guidance |
| Which sessions to host | Program creation |

## First 30 seconds

The first 30 seconds should feel like:

**There is a place for me here, and the next good match is within reach.**

The user should not feel like they entered an empty marketplace or a complex club portal.

The first experience should answer:

| Question | Product answer |
| --- | --- |
| Can I play soon? | Show a small number of relevant upcoming matches, not a giant list. |
| Will I fit? | Show level range and confidence language. |
| Can I trust this? | Show confirmed players, reliability signals, and clear rules. |
| Where am I anchored? | Ask for or infer home venue early. |
| What is expected of me? | Make commitment, payment, and cancellation visible. |

## Core daily/weekly habit

The habit is not daily scrolling.

The habit is weekly return to play.

| Moment | Habit |
| --- | --- |
| Early week | Check recommended matches and open-play sessions. |
| Before match | Confirm players, rules, and arrival. |
| After match | Complete quick feedback and receive recap. |
| Later | Join the next match, squad, clinic, or coach session. |

The product should earn attention by helping people play again, not by keeping them inside the app.

## Data captured from day one

| Data | Why it matters |
| --- | --- |
| Home venue | Anchors local network effects. |
| Availability | Helps match supply to real demand. |
| Self-rated level | Seeds early matching. |
| Preferred side / style | Improves match and partner fit over time. |
| Match join intent | Shows what people want before they play. |
| Payment completion | Confirms commitment. |
| Cancellation timing | Feeds reliability. |
| Attendance | Builds trust graph. |
| Match result | Helps skill confidence. |
| Balance rating | Measures quality, not just completion. |
| Peer reliability feedback | Detects trust problems. |
| Improvement tag | Creates coaching pathway. |
| Recommendation follow-through | Shows what next actions actually work. |

## What should not be included in V1

| Do not include | Why |
| --- | --- |
| Generic activity feed | It distracts from real play and is hard to seed. |
| Broad sports support | Padel-specific trust and level dynamics matter more. |
| Full club ERP | Too complex and not the core behavior model. |
| Pro-shop/POS/accounting | Commodity club operations. |
| Full AI video analysis | Needs data and capture reliability first. |
| Public leaderboards by default | Can create ego and mismatch before trust exists. |
| Complex tournaments | Open play and recurring squads are the first behavior model. |
| Infinite court browsing | V1 should recommend the right match, not create decision fatigue. |
| Hidden ratings | Opaque scores break trust. |

## Onboarding should optimize for

Onboarding should optimize for **first trusted match**, not profile completeness.

| Onboarding question | Why ask |
| --- | --- |
| Where do you want to play? | Creates local context. |
| When do you usually play? | Creates availability signal. |
| How would you describe your level? | Seeds first match fit. |
| What kind of match do you want? | Social, competitive, practice, mixed, clinic. |
| Have a coach or need validation? | Creates path to trusted level. |

Onboarding should avoid long personality tests, heavy profile building, and social graph import before value is clear.

## What creates retention

| Retention source | Why it works |
| --- | --- |
| Better match quality | Good matches create immediate desire to return. |
| Familiar players | Repeated recognition creates belonging. |
| Reliability reputation | Players do not want to lose earned trust. |
| Level confidence | Progress becomes visible and useful. |
| Post-match recap | The match has memory and meaning. |
| Next action | The product tells the player how to continue. |
| Venue rhythm | Recurring sessions create weekly routine. |
| Coach pathway | Improvement becomes guided rather than vague. |

## What creates emotional attachment

Emotional attachment comes from being seen accurately over time.

| Product memory | Emotional meaning |
| --- | --- |
| "You show up" | I am trusted. |
| "These players fit you" | I belong here. |
| "This match was balanced" | My time was respected. |
| "Your level confidence changed" | I am improving. |
| "This coach knows your pattern" | I am understood. |
| "This venue is becoming your place" | I have a sporting home. |

The product should make the user feel grounded confidence, not hype.

## What creates long-term defensibility

| Defensible asset | Why it matters |
| --- | --- |
| Local match graph | Records who plays well with whom in real venues. |
| Reliability graph | Built from verified commitment and attendance over time. |
| Skill confidence graph | Combines outcomes, feedback, and coach validation. |
| Venue demand map | Reveals what programs should exist at what times and levels. |
| Coach validation layer | Adds trusted human labels competitors cannot scrape. |
| Recap-to-return history | Shows which insights create actual repeat play. |

The defensibility is not the interface.

It is the memory of the local padel community.
