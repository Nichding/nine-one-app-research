# First User Experience

## First experience principle

The first experience should not feel like software setup.

It should feel like arriving at a good club:

**clear, welcoming, specific, and quietly confident.**

The user should understand within 30 seconds:

1. There are real matches near them.
2. The system cares about level fit.
3. Showing up matters.
4. This is about playing better with people, not browsing listings.

## First 30 seconds

| Time | Experience | Feeling |
| --- | --- | --- |
| 0-5 seconds | Nine One asks where the player wants to play or detects nearby partner venues | "This is local." |
| 5-10 seconds | Player sees a small number of upcoming suitable matches or open-play sessions | "There is a way in." |
| 10-20 seconds | Each match shows level range, confirmed players, reliability cues, and clear commitment rules | "This feels safe to join." |
| 20-30 seconds | User is asked only the minimum needed to improve fit: level, availability, preferred venue | "This is helping me find the right match." |

No empty feed. No generic homepage. No broad feature tour.

## First-session promise

The app should answer the user's silent questions:

| Silent question | Product response |
| --- | --- |
| Am I good enough? | Show matches by level range with confidence language. |
| Will this be awkward? | Show who is confirmed, what kind of match it is, and how commitment works. |
| Will people show up? | Show reliability and payment/confirmation state. |
| What happens after? | Promise a short match recap and next recommendation. |
| Is this just another booking tool? | Lead with match fit and player trust, not court inventory. |

## First action

The ideal first action is:

**Join or save a recommended open match.**

If no match is available, the fallback should not be a dead end. It should capture demand:

| If no immediate match exists | Product should |
| --- | --- |
| Wrong time | Ask preferred times and add user to demand list. |
| Wrong level | Invite player to level validation or suitable clinic. |
| Wrong venue | Show nearest partner venue and collect home venue interest. |
| Match full | Offer waitlist and substitute role. |

Every non-booking should still strengthen the network.

## Onboarding sequence

| Step | Question | Keep / avoid |
| --- | --- | --- |
| 1 | Where do you want to play? | Keep. Anchors venue and network. |
| 2 | When can you usually play? | Keep. Creates demand signal. |
| 3 | What is your level? | Keep, but allow "not sure." |
| 4 | What kind of match do you want? | Keep: social, competitive, practice, clinic. |
| 5 | Add photo/name | Keep for trust before joining. |
| 6 | Long bio, interests, social import | Avoid. Not needed for first trusted match. |

## First match journey

| Stage | Product behavior | Emotional goal |
| --- | --- | --- |
| Join | Clear price, rules, cancellation, level, and player status | Trust |
| Before play | Reminder with venue, time, players, and expectations | Calm readiness |
| After play | Ask for result, balance, reliability, improvement area | Reflection |
| Recap | Show match quality, level confidence, reliability, next action | Grounded confidence |
| Return | Recommend another match, squad, clinic, or coach validation | Continuity |

## What the first experience must avoid

| Avoid | Why |
| --- | --- |
| Giant map of courts | Feels like a commodity marketplace. |
| Empty community feed | Signals no one is here. |
| Long onboarding | Blocks the user's real intent. |
| Gamified badges too early | Feels shallow before trust exists. |
| AI claims before data | Creates skepticism. |
| Hidden fees or unclear rules | Breaks commitment. |
| Showing too many low-fit matches | Creates anxiety and decision fatigue. |

## First experience success metric

| Metric | Meaning |
| --- | --- |
| First trusted match intent | User joins, saves, waitlists, or gives availability for a recommended match. |
| Profile trust completion | User adds enough identity and level data to be acceptable to other players. |
| Match-fit confidence | User understands why a match is recommended. |
| No dead-end rate | Users without immediate matches still leave useful demand data. |
