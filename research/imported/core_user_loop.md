# Nine One Core User Loop

## Core loop

The MVP loop should be:

**Join match -> play -> rate match -> receive recap -> improve profile -> get better next match.**

This loop creates repeated player value and gives Nine One proprietary data that improves future matches.

## Player loop

| Step | User action | Product response | Data captured | Strategic value |
| --- | --- | --- | --- | --- |
| 1 | Player sets level, availability, home venue, and preferences | Nine One recommends suitable open matches | Self-rated level, preferred times, venue affinity, side preference | Starts personalization before enough match history exists. |
| 2 | Player joins an open match | Payment is confirmed and slot is locked | Intent, price tolerance, level range, booking conversion | Creates reliable demand signal. |
| 3 | Player attends and completes match | Match moves to completed state | Attendance, no-show/cancellation, court/time/context | Builds reliability graph. |
| 4 | Players submit quick feedback | Nine One asks for result, balance, player reliability, and optional improvement area | Match outcome, perceived balance, peer trust, skill signals | Improves matchmaking and level confidence. |
| 5 | Player receives recap | Shows match quality, level confidence, reliability impact, next match, and one improvement recommendation | Engagement with recap and recommendations | Creates retention habit. |
| 6 | Player books next action | Player joins next match, clinic, or coach session | Follow-through on recommendation | Converts insight into revenue and stronger data. |

## Venue loop

| Step | Venue action | Product response | Data captured | Strategic value |
| --- | --- | --- | --- | --- |
| 1 | Venue publishes open-play templates | Nine One surfaces matches to suitable players | Court supply, time slots, level bands, pricing | Creates repeatable supply. |
| 2 | Players fill slots | Venue sees fill status and payment state | Demand by time, level, price, venue | Helps venues optimize programming. |
| 3 | Match completes | Venue sees completion and reliability outcomes | Utilization, no-show risk, player quality | Builds trust and venue ROI. |
| 4 | Nine One recommends next program | Venue can run clinic, ladder, social night, or off-peak offer | Segment behavior and conversion | Moves from booking tool to growth system. |

## Coach loop

| Step | Coach action | Product response | Data captured | Strategic value |
| --- | --- | --- | --- | --- |
| 1 | Coach validates player level | Profile gains coach-validated confidence | Coach rating, skill band, notes | Improves trust in matching. |
| 2 | Coach tags improvement area | Recap recommends clinic/drill/session | Weakness category, intent, conversion | Connects match data to monetization. |
| 3 | Coach hosts clinic/open play | Nine One fills sessions with suitable players | Coach demand, attendance, repeat behavior | Builds coach-side network effect. |

## Loop design principles

| Principle | MVP implementation |
| --- | --- |
| Feedback must be lightweight | Post-match feedback should take under 30 seconds. |
| Progress must be visible | Show confidence bands and why recommendations changed. |
| Trust must be earned | Reliability should update from attendance and peer feedback, not only self-reported claims. |
| Every recap needs a next action | Recommend one match, one clinic, or one improvement focus. |
| Venue value must be obvious | Show fill rate, repeat players, no-shows avoided, and off-peak utilization. |

## Anti-loops to avoid

| Anti-loop | Why to avoid |
| --- | --- |
| Browsing many empty matches | Destroys marketplace trust before liquidity exists. |
| Generic social feed with no activity | Hard to seed and easy to ignore. |
| Long onboarding quiz | Reduces first-booking conversion. |
| Opaque rating changes | Competitor complaints show rating trust is fragile. |
| Recaps with stats but no next action | Interesting data without behavior change does not compound. |
